# تسيير مالية العمارة — Building Finance Manager

تطبيق ويب كامل (HTML + CSS + JavaScript فقط، بدون أي خادم) لتسيير أداءات
السكان، المصاريف، المتأخرات، الوصولات، وكشوف الحساب نصف السنوية للعمارة.

## نشر التطبيق على GitHub Pages

1. أنشئ مستودع (repository) جديد على GitHub، مثلا باسم `imm30-finance`.
2. ارفع جميع ملفات هذا المجلد (`index.html`, `css/`, `js/`) إلى جذر المستودع.
3. اذهب إلى **Settings → Pages**، واختر **Deploy from a branch**، ثم الفرع
   `main` والمجلد `/root`، واحفظ.
4. بعد دقيقة أو دقيقتين سيصبح التطبيق متاحا على رابط من نوع:
   `https://<username>.github.io/imm30-finance/`
5. أضف الرابط كعلامة مرجعية (bookmark) على هاتف أو حاسوب من يستعمل التطبيق.

لا حاجة لأي حساب أو خدمة مدفوعة. كل البيانات تبقى محليا داخل متصفح كل جهاز
(LocalStorage) — لذلك من المهم جدا استعمال **تصدير نسخة احتياطية** بانتظام
(صفحة الإعدادات)، خصوصا قبل تغيير الهاتف/الحاسوب أو مسح ذاكرة التصفح.

## البيانات الأولية

قمت بتحليل الكشف المرفق (يناير - يونيو 2026) واستخرجت منه تلقائيا:
- أسماء وأرقام الشقق الـ16.
- تاريخ آخر شهر مُؤدى لكل شقة (ومنه تُحسب المتأخرات تلقائيا من الآن فصاعدا).
- المصاريف المسجلة في تلك الفترة، مصنفة حسب النوع.
- رصيد افتتاحي محسوب بحيث يُعطي بالضبط الرصيد الحالي المذكور في الكشف
  الأصلي (7424.00 درهم).

هذه البيانات هي **نقطة انطلاق قابلة للتعديل بالكامل** من صفحة الإعدادات
(الأسماء، تاريخ صلاحية الوصل لكل شقة، الوجيبة الشهرية...). التطبيق لا يعيد
إنتاج الكشف يدويا — بعد هذه النقطة، كل الحسابات تصير تلقائية بالكامل.

## كيف تعمل الحسابات (بدون أي إدخال يدوي للمتأخرات)

كل شقة لها حقل واحد فقط يُخزَّن فعليا: **"الوصل صالح إلى غاية شهر كذا"**.
كل شيء آخر يُحسب من هذا الحقل تلقائيا:

- **المتأخرات** = عدد الأشهر الفاصلة بين هذا الشهر واليوم × 50 درهم.
- عند تسجيل أداء: يُقسَّم المبلغ على 50، وتتقدّم صلاحية الوصل بعدد الأشهر
  الناتجة. أي باقي (مثلا أداء 70 درهم) يُحفظ كرصيد يُضاف تلقائيا للأداء
  الموالي، فلا يضيع أي درهم.
- **رصيد العمارة** = رصيد افتتاحي + مجموع كل الأداءات - مجموع كل المصاريف،
  محسوب دائما، بدون أي جمع يدوي.

## اقتراحات لتحسين المسار المحاسبي (بناء على تحليل الكشف الأصلي)

هذه ملاحظات اختيارية، ليست منفذة إجباريا، لكنها فرص تحسين لاحظتها:

1. **فصل "تاريخ الأداء" عن "الفترة المُغطاة"**: حاليا الكشف الأصلي يربط
   المتأخرات بشهر واحد بدل تتبع كل دفعة على حدة. النظام الجديد يحتفظ
   بسجل كامل لكل دفعة (بالضبط لتحديد "من دفع كم ومتى")، وهذا يسهل حل أي
   خلاف مستقبلي حول دفعة معينة.
2. **رصيد مرحّل واضح بين الفترات**: بدل حساب الرصيد يدويا كل ستة أشهر،
   التطبيق يحسبه لحظيا، ويمكن معاينة "الرصيد كما كان في تاريخ معين" لأي
   تقرير سابق دون الحاجة لإعادة الحساب.
3. **تنبيه استباقي للمتأخرات الكبيرة**: يمكن مستقبلا (تحسين اختياري) إضافة
   ترتيب تلقائي للشقق حسب حجم المتأخرات في لوحة القيادة، لتسهيل المتابعة
   مع الحالات الأكثر إلحاحا أولا.
4. **فصل مصاريف الطوارئ عن المصاريف الدورية**: يمكن مستقبلا إضافة علامة
   "مصروف استثنائي" (كإصلاح الأنترفون) مقابل "مصروف دوري" (كأجرة السيدة
   المنظفة)، لتحليل الميزانية السنوية المتكررة بمعزل عن النفقات الاستثنائية.
5. **توحيد الوجيبة الشهرية مع إمكانية استثناء شقة**: التطبيق يفترض 50 درهم
   لكل الشقق. إن كانت هناك شقة بوجيبة مختلفة مستقبلا، يُنصح بإضافة حقل
   "وجيبة خاصة" لكل شقة بدل الوجيبة الموحدة (غير منفذ حاليا لأن الكشف
   الحالي يفترض توحيد الوجيبة).

## بنية الملفات

```
index.html              الهيكل العام وكل الصفحات (لوحة القيادة، الأداءات، المصاريف، التقارير، الإعدادات)
css/style.css            التصميم، دعم الاتجاه من اليمين لليسار، وتنسيق الطباعة
js/data.js                نموذج البيانات، التخزين المحلي، ومحرك الحسابات
js/utils.js               تنسيق التواريخ والمبالغ بالعربية
js/print-templates.js     توليد الوصولات وكشوف الحساب القابلة للطباعة/الحفظ كـ PDF
js/app.js                 منطق الواجهة والتفاعل
```

## طباعة الوصولات والتقارير كملفات PDF

بدل استعمال مكتبة خارجية (قد لا تدعم العربية بشكل جيد)، يعتمد التطبيق على
خاصية **الطباعة المدمجة في المتصفح** (Ctrl+P أو زر "طباعة / حفظ PDF")، ثم
اختيار **"حفظ كـ PDF" / Save as PDF** من نافذة الطباعة. هذا يضمن عرض
النصوص العربية بشكل صحيح دائما، ولا يتطلب أي اتصال بالإنترنت.
